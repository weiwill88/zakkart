文档日期：2026-04-07

## 1. 项目分期边界

### 1.1 一期目标

一期的目标不是做完整 ERP，也不是一次性把库存、财务、FBA、经营分析全部做完。
一期只聚焦一条最核心、最常发生、最需要尽快跑通的供应链履约闭环：

`下单 → 合同签署 → 生产跟踪 → 验货 → 发货物流 → 组装厂收货 / 货代确认`

一期当前覆盖的主要能力如下：

- PC 端主数据管理：产品、SKU、BOM、供应商、地址
- 合同管理：生成合同、供应商确认、导出 Word、上传已签 PDF
- 生产与验货：供应商标记生产完成，甲方提交验货结果
- 发货与物流：创建发货单、数量确认、司机扫码、在途、收货确认、货代确认
- 最小库存底层：仅支撑“收货入库”这一条能力，不扩展为完整库存系统
- 通知基础能力：合同、发货、司机、货代相关通知写入系统并可查看/已读

### 1.2 二期目标

二期主要聚焦“组装厂作业 + 库存治理”，是在一期稳定后继续补强，不属于本次验收范围：

- 组装/包装/出库完整作业链
- 更完整的库存管理界面
- 盘点、库存调整、库存治理
- 退货返修更完整流程
- 更多异常处理闭环

### 1.3 三期目标

三期主要聚焦“经营视图与优化增强”，也不属于本次验收范围：

- 驾驶舱和经营分析
- 更完整报表
- FBA / Amazon 数据对接
- 更复杂的预警和智能提醒
- 财务结算与经营视图

建议部署到 Vercel 或其他 HTTPS 静态托管平台。
当前前端已经是标准 Vite 构建产物，可以直接部署。

建议口径：

当前小程序在测试阶段使用的是手机号 + 验证码登录链路，不强依赖正式 `getPhoneNumber` 流程，因此当前可先用于联调和试运行测试。

司机无需注册登录，只需打开发货单生成的专属链接或二维码，即可完成 Step1 / Step2。

## 4. 整体业务流转逻辑

本系统的一期核心不是“看页面”，而是“跑闭环”。

### 4.1 总体流程

1. 甲方在 PC 端下单，系统按 BOM 拆解并生成合同草稿
2. 甲方完善合同内容，推送给供应商确认
3. 供应商在小程序确认合同内容
4. 甲方线下去电子签平台发起签约
5. 签署完成后，甲方将已签 PDF 上传回系统归档
6. 甲方在合同下录入结构化交付批次
7. 供应商在小程序查看待生产任务，并标记“已生产待验货”
8. 甲方安排验货，并在 PC 端提交验货结果
9. 验货通过后，该批次进入“已验货待发货”
10. 供应商或组装厂创建发货单
11. 如果实际发货数量和计划不一致，甲方先确认差异
12. 司机通过同一个 H5 链接先做 Step1，再做 Step2
13. 货物进入“在途”
14. 如果目的地是组装厂，则由组装厂在小程序确认收货并执行最小入库
15. 如果目的地是货代，则由甲方跟单员在 PC 端确认“到达货代”，只做状态确认，不做入库

### 4.2 合同后为什么不一定立即出现“待生产”

合同里只有正文内容还不够。
系统要出现生产入口，必须先录入结构化交付批次，也就是写入 `delivery_batches`。

测试时必须确认：

- 合同详情里已新增交付批次
- 批次日期已填写
- 配件数量已填写

只有这样，供应商小程序才会出现：

- 待标记已生产
- 待验货
- 待发货

### 4.3 司机 H5 的两步逻辑

司机侧不是两个二维码，而是同一个链接复用：

1. Step1：登记司机姓名、手机号、车牌号
2. Step2：在装货现场核对清单、手写签名、上传装车照、获取 GPS、确认提货

如果司机没有提前做 Step1，也可以到现场连续完成 Step1 + Step2。

### 4.4 收货与货代确认的差异

这两个动作一定要分清：

- 发往组装厂：确认收货时要写入最小库存
- 发往货代：确认到达货代时只更新状态，不写库存

## 7. 甲方验收主清单

以下建议按顺序验收。

### 7.1 主数据与基础配置

- 产品列表、产品详情、SKU、BOM 是否可正常查看和维护
- 供应商资料、地址、成员管理是否可用
- 供应商成员权限是否能正确保存

### 7.2 合同链路

- PC 端能否下单生成合同草稿
- 合同详情是否能编辑和录入交付批次
- 是否能推送给供应商确认
- 供应商小程序是否能查看并确认合同
- 是否能导出 Word
- 是否能上传已签 PDF 并归档

### 7.3 生产与验货链路

- 供应商是否能看到待生产任务
- 标记已生产后，甲方是否能在 PC 端看到待验货项
- 甲方提交验货结果后，状态是否正确推进到“待发货”
- 验货附件是否能正常查看

### 7.4 发货创建链路

- 供应商是否能从待发货明细发起发货
- 发货地址与目的地是否按业务规则正确过滤
- 实际发货数量与计划不一致时，系统是否要求填写原因
- 数量异常是否需要甲方确认后司机才能继续

### 7.5 司机 H5 链路

- PC / 小程序中是否能拿到司机链接
- Step1 是否能成功登记司机信息
- Step2 是否能成功提交签名、装车照、GPS
- 发货详情中是否能看到司机姓名、车牌、电话、签名、照片、GPS

### 7.6 收货链路

- 组装厂是否能看到待收货列表
- 确认收货后，发货单是否变为 `ARRIVED`
- 收货数量差异是否能记录
- 收货确认后是否写入最小库存
- `inventory_change_logs` 是否同步记录入库日志

### 7.7 货代链路

- 跟单员是否能上传货代入库指示单
- 单据是否能关联一个或多个发货单
- 点击“确认到达货代”后，关联运单是否统一进入到达状态
- 货代确认后是否没有写入库存

### 7.8 通知与可见性

- 小程序通知页是否能看到真实通知
- PC 通知中心是否能看到真实通知
- 已读/未读是否正常
- 通知点击跳转是否正确

### 7.9 权限回归

建议至少验证以下四类账号：

1. 负责人账号
应可看到合同、生产、发货、我的、成员管理全部能力

2. 只有合同权限的成员账号
应只能看到合同相关能力，不应看到生产、发货操作入口

3. 只有发货权限的成员账号
应能创建/查看发货，不应看到无关模块

4. 只有成员管理权限的成员账号
应能进入成员管理，但不应拥有发货或生产操作权限

## 8. 重点验收场景

### 场景 A：普通供应商发货到组装厂

建议按以下顺序完整跑一遍：

1. 管理员生成合同并录入批次
2. 供应商确认合同
3. 管理员上传已签 PDF
4. 供应商标记已生产
5. 管理员验货通过
6. 供应商创建发货单
7. 管理员确认数量差异
8. 司机完成 Step1 / Step2
9. 组装厂确认收货
10. 校验库存已入库

### 场景 B：组装厂或直发供应商发货到货代

建议按以下顺序完整跑一遍：

1. 发货单创建
2. 司机完成 Step1 / Step2
3. 跟单员上传货代单据
4. 跟单员确认到达货代
5. 校验状态更新正确
6. 校验未写入库存
